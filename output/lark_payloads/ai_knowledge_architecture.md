# AI知识库架构设计与Agent落地方案

## 一页结论

这份资料的核心判断是：公司需要建设的不是一个把所有资料都塞进去的“巨型向量库”，而是一套统一 AI Context Platform。原始数据仍保留在各自系统中，向量库只作为检索索引层；数仓、火山引擎、客服系统等实时或强一致数据应通过受控 Tool/API 查询。

第一阶段建议采用：Hermes Agent 作为团队 AI 中枢，自研 Knowledge MCP / Retrieval API 作为企业知识库入口，飞书作为第一知识源和第一交互入口；OpenClaw 先不同时作为主系统，后续作为 IM 网关或特定自动化 Agent 补充接入。

## 推荐架构

```text
飞书知识库 / IM / 客服 / 爬虫 / 数仓 / 火山引擎
        ↓
数据接入层：自研 Connector + SeaTunnel / Airbyte / n8n
        ↓
原始数据与版本存储：OSS/MinIO + PostgreSQL Metadata
        ↓
知识加工流水线：解析、清洗、去重、脱敏、ACL、切块、摘要、Embedding、索引
        ↓
混合检索与工具层：OpenSearch/Qdrant/pgvector + SQL Gateway + API Tools
        ↓
统一 Retrieval API / Knowledge MCP：身份校验、ACL、引用溯源、审计日志
        ↓
Hermes Agent / Dify / RAGFlow / 自研助手
        ↓
飞书、钉钉、微信、Web、日报周报、客户报告
```

## 数据源处理原则

| 数据源 | 第一阶段处理方式 | 关键注意事项 |
| --- | --- | --- |
| 飞书知识库 | 作为主数据源同步，按文档结构切块 | 保留路径、作者、更新时间、原始链接、版本、权限 |
| IM 群聊 | 保留原始消息，提炼决策、待办、故障方案、Q&A | 不要把每句话直接向量化；结论要晋升为正式知识 |
| 数仓 | 指标口径、字段说明、常用 SQL 进知识库；实时数值走 SQL Tool | 只读、SELECT、限时限量、脱敏、禁止直连生产主库 |
| 火山引擎 | 稳定文档进知识库，实时任务状态和用量走 API Tool | 不要把实时状态定时复制成向量 |
| 客服/爬虫/第三方 API | 记录来源、生命周期、可信度、客户 ID、分类 | 检索分数要综合语义、关键词、可信度、新鲜度 |

## 权限与安全底线

1. ACL 必须在检索前过滤，不能先搜出敏感内容再靠 Prompt 要求模型保密。
2. 每个 chunk 至少携带用户、部门、群组、知识空间、客户、项目、敏感级别等权限元数据。
3. 原文删除、用户离职、部门变更后，索引权限必须同步失效。
4. 数仓默认只开放聚合结果表、安全宽表和只读视图，新增分析需求由数分补安全视图。
5. 飞书写入、数据库查询、外部 API 调用、报告发送都需要审计日志。
6. 公共团队 Agent 与敏感经营分析 Agent 建议使用不同凭证、不同数据源，必要时物理隔离。

## 平台选型建议

| 组件 | 建议定位 |
| --- | --- |
| Hermes Agent | 主 Agent Runtime，负责入口、意图判断、工具调度、群聊回复、定时任务、子 Agent |
| OpenClaw | 后续作为补充的 IM 网关或特定自动化 Agent；第一版不建议与 Hermes 双主运行 |
| Knowledge MCP | 自研核心知识入口，统一处理检索、ACL、引用、审计、日志 |
| RAGFlow | 文档解析、知识检索、RAG 验证中心 |
| Dify | AI 应用和工作流门户，可选接入自建 Retrieval API |
| PostgreSQL + pgvector | 第一阶段低成本验证，适合百万级以下 chunk |
| OpenSearch / Qdrant | 生产化后做混合检索、payload filter、多租户和 ACL 预过滤 |
| SeaTunnel / Airbyte / n8n | 分别承担数据库/CDC、标准 API Connector、零散 API/事件工作流 |
| Langfuse / Ragas | 调用链观测与 RAG 离线评估 |

## 第一版 PoC 范围

建议先用 3 到 5 个真实任务比较 Hermes 与 OpenClaw，任务保持相同数据和模型：

1. 查询“这个服务怎么发布？”验证飞书知识库 RAG。
2. 查询“上次类似故障怎么解决的？”验证故障记录与引用溯源。
3. 查询“昨天群里最后决定了什么？”验证 IM 决策提炼。
4. 查询“今天哪个客户收入最高？”验证只读 SQL Tool。
5. 生成“某客户退费率异常归因”验证台账、指标和归因 Skill。

第一阶段只接入：飞书正式文档、指定项目群、已确认故障记录/技术决策、数仓表结构与指标口径、一个火山或业务 API。

## 近期行动项

1. 基础设施：确定 Linux 服务器，规划公共环境与敏感环境是否分离，迁移本地机器人/定时任务。
2. 平台 PoC：分别部署 Hermes Agent 与 OpenClaw，按同一任务集评估正确率、稳定性、权限、渠道接入和 Token 成本。
3. 数据准备：整理 AI 可访问的结果表、宽表、字段注释、指标口径和禁止字段。
4. 知识准备：扫描飞书知识库，为文档增加项目、客户、部门、权限标签，区分公共 SOP 和客户私有资料。
5. 业务优先级：从运营日志自动入库、每日播报预警、退费率异常归因中先选 1 到 2 个闭环。
6. Skill 沉淀：把成熟 SQL/Python/分析流程写成团队可复用 Skill，包含输入参数、输出格式、异常规则和人工复核要求。

9. 正式纪要中客户和渠道名是否保留，还是先匿名为客户 A、渠道 B。

1. 飞书知识库
飞书应该作为第一阶段的主数据源，因为内容质量通常高于群聊。

同步方式建议：

首次遍历 Wiki 空间和节点树。

获取每篇文档的结构化 block。

转为统一 Markdown/JSON。

保存文档版本、路径、作者、更新时间、原始链接和权限。

使用事件订阅做增量同步。

每天再进行一次全量校验，防止漏事件。

飞书开放平台已经提供 Wiki、文档、节点遍历和事件订阅能力，因此不需要通过浏览器爬页面。

统一后的文档可以长这样：

{
  "source_type": "feishu_wiki",
  "source_id": "wikcn_xxx",
  "title": "生产环境发布流程",
  "content": "...",
  "space_id": "xxx",
  "path": "/运维/SRE/发布流程",
  "author_id": "ou_xxx",
  "updated_at": "2026-06-26T10:00:00+08:00",
  "source_url": "...",
  "version": 17,
  "checksum": "sha256:...",
  "acl": {
    "users": [],
    "departments": ["sre"],
    "visibility": "department"
  }
}
飞书文档切块不要只按字数
优先按文档结构：

文档标题
  └── 一级标题
       └── 二级标题
            └── 段落 / 表格 / 代码块
每个 chunk 都携带：

文档标题

标题路径

原文链接

更新时间

权限

上下文摘要

这样 AI 引用时能回答：

来源：运维知识库 / 生产发布 / A-B 切流
最后更新：2026-06-25

而不是吐出一块来历不明的语义砖头。

2. 飞书、钉钉、微信等 IM
群聊是最容易把知识库弄脏的地方。

不要把每句话直接向量化，例如：

好的
收到
我看看
应该行
@小王 这个呢
这些内容量大、信息密度低，还会淹没正式文档。

建议建立两层：

原始消息层
保留完整消息，仅用于审计和重新处理：

group_id
message_id
thread_id
sender_id
timestamp
text
attachments
reply_to
mentions
deleted
知识提炼层
按 Thread 或时间窗口聚合，再提取：

{
  "type": "decision",
  "title": "生产渠道号改为 U7kdtRdD",
  "summary": "生产构建使用正式渠道号，预发布仍使用原渠道号。",
  "participants": ["张三", "李四"],
  "decided_at": "2026-06-25T19:30:00+08:00",
  "evidence_message_ids": ["om_x1", "om_x2"],
  "project": "fulian-admin-ui",
  "status": "confirmed"
}
重点提炼四种知识：

决策 Decision

待办 Action Item

故障与解决方案 Incident/Solution

问题与答案 Q&A

飞书支持消息事件订阅，但“读取关联群内全部消息”属于敏感权限，因此 IM 权限必须单独审批并严格保留 ACL。

另外，群聊里的结论最好定期“晋升”为正式知识：

群聊讨论
   ↓
AI 生成决策草稿
   ↓
人工确认
   ↓
写回飞书知识库
   ↓
正式知识权重大于群聊知识
这能避免 AI 把群友一句“我感觉可能是 Redis”当成公司技术圣旨。

3. 数仓
不要把数仓明细全部向量化。

数值会变化，Embedding 也不擅长精确计算。应该拆成两部分：

进入向量知识库的内容
表结构

字段含义

指标口径

数据血缘

常用 SQL

报表说明

数据负责人

更新时间及 SLA

例如：

{
  "table": "dws_order_daily",
  "metric": "有效订单数",
  "definition": "支付成功且未全额退款的订单数",
  "sql_example": "SELECT ...",
  "owner": "数据组",
  "security_level": "internal"
}
实际数值
由 AI 生成 SQL，经 SQL Gateway 查询实时数据：

用户问题
  ↓
检索指标口径和表结构
  ↓
生成 SQL
  ↓
SQL 审计 / 白名单 / LIMIT
  ↓
只读账号执行
  ↓
返回结果并解释
所以数仓是 Tool，不是普通文档知识库。

必须加这些护栏：

只允许 SELECT
限制扫描行数和执行时间
屏蔽敏感列
基于用户身份控制表权限
记录生成 SQL 和查询结果
禁止直接连接生产主库
SeaTunnel适合 JDBC、批量同步和数据库 CDC，官方连接器支持 MySQL、PostgreSQL、Oracle、SQL Server 等数据源。

4. 火山引擎
这里要分两种数据：

稳定知识
例如产品文档、配置说明、内部使用规范，可以同步进知识库。

实时状态
例如：

当前任务状态

资源使用情况

作业结果

最新账单

模型调用量

不要定时复制成向量，直接封装成 API Tool：

get_volcengine_job_status(job_id)
get_resource_usage(project_id)
query_model_usage(start_time, end_time)
向量库负责告诉 AI“应该调用哪个工具以及参数是什么”，API 负责返回真实现状。

5. 外部爬虫、客服和第三方 API 落表
这一类最重要的是数据可信度和生命周期。

建议每条记录增加：

source_system
source_record_id
source_url
collected_at
updated_at
valid_from
valid_to
checksum
reliability_level
customer_id
data_classification
外部信息还应设置权重：

内部正式文档        1.00
人工确认的决策记录  0.95
工单解决记录        0.90
群聊总结            0.75
原始群聊消息        0.50
外部爬虫内容        0.40
检索结果不只是看相似度，还应综合：

最终分数 =
语义相关性
+ 关键词相关性
+ 来源可信度
+ 新鲜度
+ 人工确认权重
权限是整个项目的地基
这个项目最危险的情况不是 AI 答错，而是：

销售问了一个问题，AI 检索到了管理层私密群或其他部门知识库。

每个 chunk 至少要携带：

{
  "tenant_id": "company_1",
  "source_type": "feishu",
  "space_id": "sre",
  "allowed_user_ids": [],
  "allowed_department_ids": ["sre"],
  "allowed_group_ids": [],
  "visibility": "department",
  "sensitivity": "internal"
}
查询流程必须是：

用户身份
  ↓
获取部门、群组、知识空间权限
  ↓
生成 ACL Filter
  ↓
在向量搜索前过滤
  ↓
语义检索
不能先搜出秘密内容，再在 Prompt 里告诉模型“你不要说”。那相当于把保险柜打开，再拜托猫别碰鱼。

Qdrant 和 Milvus 都支持基于 metadata/payload 的检索过滤，可以用作 ACL 预过滤。

同时必须支持：

原文删除后，索引同步删除

用户离职后，权限立即失效

部门变更后，重新计算访问范围

回答展示原始引用链接

管理员能查到“为什么搜到了这段”

开源项目怎么选
RAGFlow
适合做文档解析、知识库检索和 RAG 服务，尤其是复杂 PDF、Word、表格等文档场景。它本身定位就是开源 RAG Engine。

优点：

文档解析能力比较突出

检索调试界面直观

支持关键词、向量、Rerank 等组合

适合先验证飞书文档导入后的效果

不足：

它不是完整的数据集成平台

飞书、数仓、各种三方 API 仍需要自己做同步层

权限模型通常要在外部再包一层

定位：检索和文档处理中心。

Dify
适合做 AI 应用、工作流、Agent 和最终用户入口。它提供知识库管理 API，也支持接入外部知识服务。

优点：

工作流好搭

模型接入方便

能把 SQL、火山引擎、工单 API 做成工具

可以把自建 Retrieval API 作为外部知识库

不足：

不建议把它当企业数据底座

数据同步、复杂 ACL、版本治理仍要独立实现

定位：AI 应用和工作流门户。

MaxKB
中文团队使用比较顺手，支持文档上传、网页抓取、自动切分、向量化和工作流；其开源版本使用 PostgreSQL + pgvector。

适合：

快速做内部知识问答

团队规模不大

初期数据量有限

希望部署简单

但要注意它使用 GPLv3，后续若涉及深度二次开发、分发或商业集成，需要让法务或研发确认许可证边界。

FastGPT
国内生态成熟，提供数据处理、RAG 检索和可视化 Flow 编排，适合快速搭建知识问答和业务工作流。

它和 Dify 的角色比较接近，通常二选一即可，不建议两套都上，否则很快会收获两套模型配置、两套知识库和双倍的运维快乐。

LlamaIndex / Haystack
这两个更像开发框架，不是开箱即用产品。

LlamaIndex提供数据连接、索引、检索和查询接口；Haystack强调可组合、透明、可控的生产级 Retrieval Pipeline。

适合：

自己开发统一 Retrieval Service

对 ACL、路由、Rerank 有精细要求

数据源多且变化复杂

后面需要做多阶段检索、GraphRAG、Text-to-SQL

预警时要自动关联项目负责人，并在飞书或钉钉中 @ 对应人员。

理想结果类似：

渠道 A 今日退费率升至 28%，较过去 7 日均值上升 9 个百分点。

主要异常集中在：
1. 新增手机号查询标签用户
2. 6 月 25 日上线的价格调整
3. T+1 退费人群

负责人：@某运营
建议优先核查新标签和价格配置。
2. 自然语言数据查询
让不懂 SQL 的运营人员直接问：

今天哪个企业收入最高？
某客户最近七天退费率怎么样？
命中某标签的用户支付率是多少？
AI 根据事先定义好的指标口径和结果表生成只读 SQL，并返回解释。

3. 复杂归因分析
当支付率、收入或退费率异常时，AI需要：

读取异常时间点附近的运营动作台账。

找出最近发生的配置、标签、价格或流量变化。

按命中与未命中人群拆分。

比较不同用户桶的指标变化。

判断哪个动作贡献了主要变化。

输出可解释的归因结论。

经过验证的归因方法应被封装成 Skill，后续其他客户出现同类问题时直接复用。

4. 运营日志自动整理
目标是从飞书、钉钉或人工转发的信息中提取：

时间

客户

项目

事项类型

原始内容

影响范围

下一步动作

相关负责人

原始截图或消息链接

截图需要先 OCR 或视觉识别，但必须同时保留原图和整理结果。

这里明确强调：

AI 应尽量忠实记录原话，不应自行补充客户没有说过的信息。

5. 客户项目跟进
客户从商务接触到正式上线，生命周期较长，涉及：

销售沟通

需求收集

产品方案

开发排期

上线时间

配置确认

上线后 1 小时、3 小时、7 小时、12 小时的数据反馈

阶段性客户报告

希望未来销售通过标准问题清单录入需求后，AI自动：

生成需求记录

创建项目跟进项

查询研发进度

检查配置是否生效

在关键时间点拉取数据

生成客户汇报材料

六、初步形成的 AI 架构思路
1. IM 作为主要入口和出口
大部分同事不会主动打开 Codex、开发终端或专门的 AI 网站，因此主要交互入口应是：

飞书

钉钉

必要时微信

员工通过 @ 机器人提问，结果也直接返回群聊或私聊。

对外汇报则可能输出：

PDF

图片

网页报告

简短的飞书或钉钉消息

2. 中央路由 Agent
需要一个总 Agent 判断任务类型：

简单指标查询
    → 调用固定 SQL 或脚本

异常归因
    → 调用数据分析 Agent

知识查询
    → 调用知识库 Agent

客户汇报
    → 调用报告生成 Agent

项目进度
    → 调用项目跟进 Agent
3. 专业 Agent 和可复用 Skill
不同人员管理不同领域：

数分同学维护数据分析 Agent

运营维护指标和业务规则

技术同学维护数据连接、权限和运行环境

报告 Agent 负责生成 PDF、图片或网页

项目 Agent 负责销售和上线跟进

成熟的分析流程要沉淀为 Skill：

Skill
├── 使用哪些表
├── 指标定义
├── SQL 或 Python 脚本
├── 输入参数
├── 输出格式
├── 异常判断规则
└── 人工复核要求
4. 固定脚本优先于自由发挥
对于客户复盘、收入、退款等准确性要求高的任务，不能完全让模型临场生成。

更可靠的方式是：

Agent 判断意图
    ↓
调用已经审核过的 Python / SQL 脚本
    ↓
获得确定性结果
    ↓
AI 负责解释和排版
    ↓
人工复核后对外发送
而舆情收集、信息摘要等低风险任务，可以允许模型有更多自由度。

七、数据安全方面的初步共识
初期不建议 AI 直接访问全部底层明细表。

优先方案是为 AI 提供：

只读账号

仅允许 SELECT

聚合结果表

经过脱敏的宽表

数分整理好的指标表

不包含手机号等敏感信息

客户 ID、渠道 ID 等非直接个人身份标识

可以建立类似以下的宽表：

客户
渠道
日期
收入
首次支付人数
续费人数
T+1 退费率
T+5 退费率
T+30 退费率
标签 A 是否命中
标签 B 是否命中
近期运营动作
AI优先读取这种规整结果表，而不是面对几十张原始表自由关联。

对话中仍存在一个未完全解决的矛盾：

为了分析新标签和新问题，希望 AI 能看到更细的数据。

为了安全，希望 AI 只能访问有限的结果表。

比较现实的中间方案是：

常规任务只开放结果表；新增分析需求由数分制作新的结果表或安全视图，再加入 Agent 的允许列表。

八、知识库权限思路
知识库不能只靠 Prompt 告诉 AI“不要看别人的项目”。

初步设想是按以下维度控制：

用户 ID

部门

销售负责人

客户

项目

文档 ID

知识类型

是否公共资料

例如：

公共 SOP
    → 全公司可见

客户 A 项目资料
    → 客户 A 对应销售、运营、管理人员可见

客户 B 项目资料
    → 客户 B 对应人员可见

老板汇报和敏感经营数据
    → 独立 Agent 或独立工作区
对话里有一个很明确的安全观点：

真正不允许泄露的内容，最好让普通 Agent 根本看不到，而不是让它看到了以后依靠提示词约束。

因此可能需要：

公共团队 Agent

敏感经营分析 Agent

两套 Agent 使用不同的数据源、凭证和运行空间，甚至做物理隔离。

九、平台和部署讨论
目前正在比较的主要平台应该是：

OpenClaw，语音里多次说成“小龙虾”

Hermes Agent，语音里说成“爱马仕”

Codex 作为个人开发和编写 Skill 的工具

目前还没有最终确定使用哪个平台，倾向于：

使用相同数据、任务和模型。

定义统一评价标准。

比较准确率、Token 消耗、渠道接入和 Skill 复用能力。

再选择主要平台。

部署方面倾向于：

申请一台专门的 Linux 服务器。

不依赖个人 Windows 电脑长期运行。

WSL 可以作为临时开发环境，但生产运行优先 Linux。

可能划分两个虚拟机或两个独立环境：

一个供团队公共 Agent 使用。

一个供敏感数据和个人分析使用。

将目前运行在本地的钉钉机器人、脚本和定时任务迁移到服务器。

已经放到青龙面板中的定时任务可以继续保留或逐步迁移。

十一、比较明确的落地原则
这次讨论里已经形成了几条比较清晰的原则：

不要一开始建设一个覆盖所有场景的大平台，先逐个解决高价值问题。

底层框架需要支持持续增加 Agent 和 Skill。

员工主要从飞书和钉钉使用，而不是要求所有人进入开发工具。

准确性要求高的任务使用固定 SQL、Python 和审核机制。

低风险的信息摘要和收集任务可以给予模型更多自由。

知识库、数仓和 IM 是当前最核心的三个接入点。

成熟分析逻辑要进入团队仓库，不能只保存在个人电脑。

数据权限需要在工具和数据源层控制，不能仅依赖提示词。

先开放聚合结果表，底层明细数据谨慎开放。

需要专用 Linux 环境承载团队 Agent。
